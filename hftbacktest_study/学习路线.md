# hftbacktest 学习路线

> 仓库地址：https://github.com/EisbergDu/hftbacktest
>
> 技术栈：Rust（核心引擎） + Python（策略层绑定）

---

## 整体架构

```
数据层 (backtest/data/)
        ↓
订单簿 (depth/)          回测引擎 (backtest/mod.rs)
        ↓                        ↓
模型层 (backtest/models/)   策略层 (examples/)
  ├── queue.rs      队列位置模型（核心）
  ├── latency.rs    延迟模型
  └── fee.rs        手续费模型
```

数据流：原始 tick 数据 → 事件队列 → 订单簿维护 → 模型判断成交 → 策略 PnL 结算

## 第二阶段：核心模型

目标：理解 HFT 回测与普通回测的本质区别——**队列位置**和**延迟**。

| 文件 | 重点内容 |
|---|---|
| `hftbacktest/src/backtest/models/queue.rs` | **最重要**：队列位置模型，决定 limit order 什么时候真正成交。普通回测框架默认挂单即成交，这里精确建模你在队列中的位置 |
| `hftbacktest/src/backtest/models/latency.rs` | 延迟建模：下单延迟、确认延迟对策略的影响 |
| `examples/Probability Queue Models.ipynb` | 配合 `queue.rs`，直观理解各种队列模型的差异 |
| `examples/Impact of Order Latency.ipynb` | 量化延迟对 PnL 的具体影响，说明为什么 HFT 要死抠微秒 |
| `examples/Order Latency Data.ipynb` | 如何采集和使用真实的订单延迟数据 |

### 队列模型说明（queue.rs 的核心）

框架内置了多种队列位置估计模型：

- **`RiskAverseQueueModel`**：保守估计，假设你排在队列靠后位置，适合评估最差情况
- **`ProbQueueModel`**：概率模型，根据成交量和队列深度动态估算成交概率
- **`L3QueueModel`**：基于逐笔数据（Level-3）的精确队列追踪

## 关键概念备忘

### 为什么队列模型这么重要？
普通回测：挂上 limit order → 价格到了直接成交 ✗  
真实情况：挂上 limit order → 排在队列第 N 位 → 前面的人先成交 → 你可能没成交 ✓

队列位置决定了**成交率**，成交率直接影响**做市 PnL**，用错模型会导致回测严重高估收益。

### 延迟的影响
- **下单延迟**：你看到信号到委托到达交易所有时间差，期间行情可能已经变化
- **确认延迟**：订单状态更新不实时，导致重复下单或漏撤单
- 延迟每增加 1ms，在高频做市场景下可能损失数个 bps
